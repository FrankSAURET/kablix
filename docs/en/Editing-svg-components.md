# Editing component SVGs (and their internal schematics)

This guide explains **how to retouch a Kablix component's SVG drawing yourself**, and **how the internal schematics work / how to modify them** (the wiring shown in transparency when a component is selected).

> If you're doing this it's probably for a good reason. Please send me the corrected version or file a publication request.
---

## 1. The two families of components

| Family | SVG drawing | User-editable? |
| --- | --- | --- |
| **Built-in components** (`@wokwi/elements`: Uno, LED, resistor…) | inside the `node_modules/@wokwi/elements` package | Not directly (read-only) — see §4 |
| **Custom components** (`.kablix-part.json`) | `svg` field of the JSON file | **Yes**, freely |

The easiest way to get control over a drawing is therefore to go through a **custom component**: either by creating one, or by exporting an existing component to start from its base.

---

## 2. The golden rule: the 10 px grid

All pins must fall on a **10 px grid** (= 0.1″, the hole pitch of a breadboard and of the canvas grid). Otherwise the component won't plug in cleanly.

- Work with an **SVG document whose grid is 10 px**.
- Place every connection point (`pins[].x` / `pins[].y`) on a multiple of 10.
- `x`/`y` are in **pixels, relative to the top-left corner** of the `<svg>` tag (so they depend on `width`/`height` and the `viewBox`).

> Tip: `@wokwi/elements` boards use the physical pitch of 9.6 px; Kablix scales them up automatically (`pinScale = 10/9.6`, see `catalog.mts`). For a **custom** component, draw directly at the 10 px pitch.

## 3. Editing a custom component's SVG

### a. Get a base

- Palette → **⇪ Import (.json)** of an existing file, or
- **+ Create a part** button (palette) → the built-in editor, or
- start from a file in the [`parts/`](../../parts) folder (e.g. `picow-module.kablix-part.json`).

A `.kablix-part.json` looks like:

```json
{
  "label": "My special LED",
  "kind": "led",
  "svg": "<svg width=\"40\" height=\"56\" xmlns=\"http://www.w3.org/2000/svg\">…</svg>",
  "pins": [
    { "name": "A", "x": 10, "y": 50 },
    { "name": "K", "x": 30, "y": 50 }
  ],
  "pinRoles": { "A": "plus", "C": "minus" },
  "attrs": {}
}
```

(Full format: see the built-in help, *Component file format* section, or [`docs/en/USAGE.md`](USAGE.md).)

### b. Edit the drawing

Two methods:

1. **By hand (text)**: the `svg` field is an SVG string. Modify the colors (`fill`, `stroke`), the shapes (`rect`, `circle`, `path`)… Remember to **escape the quotes** (`\"`) since the SVG is inside a JSON string.

2. **In Inkscape / an SVG editor**:
   - set the document to a 10 px grid;
   - draw the component, place the pins on the grid;
   - **File → Save As → Plain SVG**;
   - open the `.svg`, copy the whole `<svg>…</svg>` content onto **a single line**, and paste it (escaped) into the JSON's `svg` field.

### c. Drawing constraints

- Give reasonable `width`/`height` (40 to 200 px) — that's its on-screen size. Feel free to count the squares (10 px) on Kablix for a similar component.
- **Avoid `<style>` and scripts**; prefer presentation attributes (`fill`, `stroke`, `stroke-width`…). They survive the schematic's SVG export.
- Draw a visible pad (small circle) where you declare each `pin`, to orient yourself — the connection point stays the **center** of `(x, y)`.

### d. Re-import

Palette → **⇪ Import (.json)**. The component (★) appears, ready to place. To fine-tune, **+ Create / Edit** opens the editor: the preview is zoomable (−/+) and each pin has directly **editable X / Y fields**.

## 4. What about built-in components (@wokwi/elements)?

Their SVGs live in `node_modules/@wokwi/elements/dist/esm/*-element.js` (MIT license) and are **embedded at build time**: they aren't modified from the interface. Two options:

- **Recommended**: recreate a variant as a **custom component** (§3) and use it instead.
- **Advanced** (recompilation): the Pico board is an "in-house" element ([`src/webview/composants/pico-board.mts`](../../src/webview/composants/pico-board.mts)), which reuses the `parts/picow-module` drawing and adds margins + pin names. That's the model to follow to build a custom built-in element.

## 5. Editing the internal schematics (K view)

The **internal schematic** is the wiring shown in transparency (on a white background) or the pinout (for µC boards) when you select a component and click the **K** button. It is **not** stored in the `.kablix-part.json`: it is **generated by the code**, in [`src/webview/diagram/internal-wiring.mts`](../src/webview/diagram/internal-wiring.mts) (modifying it = recompiling the extension).

### Principle

- One function per component type (`led`, `resistor`, `buzzer`, `led-bar`, `7segment`, `pushbutton`…).
- Dispatch is done by **`kind`** in `internalWiringSvg(kind, pins, attrs)`.
- The paths are in the **same coordinate frame as the `pinInfo` pins**: they therefore follow the component's rotation and flip automatically.

### Provided tools

```ts
line(a, b)                 // segment between two points {x,y}
dot(p, r?)                 // black pad (node)
mid(a, b)                  // midpoint of [a,b]
diode(from, to, catEnd)    // diode symbol A→K (bar on the `to` side if catEnd)
find(pins, 'NAME')         // position {x,y} of a pin by its name (or null)
```

### Adding / modifying a schematic

1. Write a function `myComponent(pins, attrs?): string | null` that returns an SVG fragment (assembled with `line`/`dot`/`diode`), or `null` if the expected pins are missing (`find` returns `null`).
2. Add a `case '<kind>':` to the `switch` in `internalWiringSvg`.
3. Recompile (`npm run build`). The K button appears automatically on components of that `kind` (cf. `editor.mts`, `internalWiringSvg(...)`).

> Example: `sevenSegment(pins, attrs)` reads `attrs.common` (`cathode`/`anode`) to orient its 8 diodes toward the common — a schematic can therefore **vary depending on an attribute** of the component.

## 6. Summary

| I want to… | Where to act |
| --- | --- |
| Change a custom component's **drawing** | `svg` field of the `.kablix-part.json` (§3) |
| Add my **pins** at the 10 px pitch | `pins` array of the JSON (§2) |
| Modify a **built-in** component | remake it as a custom version (§4) |
| Change the **internal schematic** (K view) | `src/webview/diagram/internal-wiring.mts` (§5) |
